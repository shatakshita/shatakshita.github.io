How I Approach Work

When a problem is unfamiliar — a new physics framework, an unfamiliar Streamlit API, a research question with no clear starting point — I start by mapping what is already known and where the gaps are, rather than guessing at a solution. The RPG project's archetype logic, for instance, started as a flowchart on paper before it became conditional branches in code.

I learn new technical material by building with it immediately rather than finishing a course first — the Task Tracker came out of needing to understand file I/O in Python, not from a tutorial assignment. This usually means early versions are rough, but it surfaces the parts of a concept that documentation alone doesn't make obvious.

In engineering and research work, I separate the question of whether something works from the question of whether it's built well — a script that produces the right output but can't be extended is, to me, half-finished. That distinction shapes how I write code, structure articles for The Redshift Asterism, and approach problems at exovista: get a working version, then rebuild the parts that won't scale.

RPG Mini Game

Python · Streamlit · State Logic

Stat-allocation game with hidden archetype discovery, built in Python and deployed as a live Streamlit app.

›

Overview

A stat-allocation game where players distribute points across Strength, Intelligence, and Charisma to unlock one of three hidden legendary archetypes, built first as a Python CLI tool and later rebuilt as a Streamlit web app.

Problem

The original CLI version worked but mixed game logic with print statements, making it impossible to reuse the logic in a different interface or test it without manually playing through every branch.

Approach

Rewrote the archetype logic as pure functions independent of display. Streamlit's session state then handles persistence between reruns, since the framework re-executes the whole script on every interaction.

Technologies

Python, Streamlit, session-state management

Challenges

Streamlit reruns the entire script on each user action — required learning st.session_state to persist stat allocations and results across reruns instead of relying on local variables.

Lessons Learned

Separating logic from presentation early made the rebuild straightforward — the hardest part was relearning state management for a framework that doesn't keep variables alive between runs by default.

Future Scope

Adding more archetypes and a stat-history view so players can see how previous distributions led to different outcomes.

Task Tracker

Python · CSV I/O · CLI Design

Console-based task manager with persistent CSV storage, focusing on file I/O, data structure, and interface design.

›

Overview

A command-line task manager that stores tasks with priorities and due dates in a CSV file, so data persists between sessions without needing a database.

Problem

Built to understand file I/O and data persistence in Python directly — a CLI to-do list is small enough to make the file-handling logic the entire point of the exercise.

Approach

Used Python's csv module with a fixed schema read into memory on launch and rewritten on every change. CRUD operations sort by priority before display; invalid input is caught before it reaches the file write step.

Technologies

Python, csv module, command-line interface

Challenges

Rewriting the entire CSV on every edit is inefficient at scale and risks data loss if the program exits mid-write — a known limitation rather than something the current version solves.

Lessons Learned

Validating input before any file operation prevents a much larger class of bugs; designing the data schema first made every later feature easier to add.

Future Scope

Migrating storage to SQLite to remove the full-file-rewrite limitation, and adding due-date reminders.
